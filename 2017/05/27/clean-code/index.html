读书笔记《代码整洁之道—程序员的职业素养》

近几天从图书馆借了好多本书，感觉逛图书馆有一种逛商场的感觉，琳琅满目的商品任我挑选，关键是想拿就拿不用结账（手动滑稽）。晚上随便拿了一本书睡前上床看，就是题目这本书。翻了翻发现和代码规范关系不大，想起可能是有另一本书详细讲解代码规范。这本书主要讲的是一名程序员的职业素养（类似于情商？）。

虽然还没有步入职业生涯（连做什么职业都没有确切规划好- -），读此书仅以开阔视野提供参考，类似经验亦可指导处理实验室团队的沟通工作。

第一章 专业主义

  • 为自己所写的代码负责，自动化测试
  • 为使自己编写的软件灵活多变，需要时刻修改
  • 投入时间，坚持学习
  • 小程序算法作热身练习或静心时间

原来曾实施过写日记练书法作为静心时间，结果一写就是一小时（- -）。

第二章 说“不”

  • 能就是能，不能就是不能，不要说”试试看“。
  • 存在矛盾时，各表异议，找到双方都能接受的方案。
  • 不要被某种荣誉光环所吸引，要坚守专业主义精神。

看书中的案例会想起在团队做美工的日子，往往自以为时间充足，结果最后灵感荒芜，为了deadline熬夜加点。在当时就没有意识到这些问题，不专业的表现。

第三章 说”是“

  • 有时候我们会轻信他人说话算话，说到做到。要学会识别缺乏责任的承诺。
  • 真正的承诺要对自己会做某件事情作清晰的事实陈述，明确完成期限。
  • 无法兑现承诺需要尽早发出预警。
  • 专业人士对自己的能力极限了如指掌，保持底线。

感觉截止时间真的十分重要，是调整做事优先级的关键因素，事事皆须明确。

第四章 编码

  • 疲劳的时候，千万不要写代码，遵循纪律原则。
  • 焦虑时可专门花一块时间处理焦虑的问题。
  • 不要进入流态区，所谓的高效可能会忽略顾全全局。
  • 音乐更可能使人滑入流态区。
  • 中断不可避免，对他人的请求保持礼貌。
  • 再找一个搭档结对编程。
  • 创造性输出依赖于创造性输入。

疲劳焦虑的状态总是那么多。（摊手）

第五章 测试驱动开发

  • TDD三项法则
  1. 编好失败单元测试之前不要编写任何代码。
  2. 只要有一个单元测试失败，不要再写测试代码。
  3. 产品恰好能够让当前失败的单元测试成功即可，不要多写。

对于测试的概念个人理解不多，不知道平常写的小程序需不需要测试。

第六章 练习

  • 卡塔练习
  • 自身经验扩展

第七章 验收测试

  • 双方准确交流细节信息是一个复杂不易的过程。
  • 编写自动化的验收测试。

第八章 测试策略

  • 自动化测试金字塔
  1. 人工探索式
  2. 系统测试
  3. 集成测试
  4. 组件测试
  5. 单元测试

第九章 时间管理

  • 会议：拒绝低回报的会议，看情况离席，须清晰议程和目标。
  • 凡是不能在五分钟内解决的争论，都不能靠辩论解决。长时间争论因为双方缺乏足够的证据，此时争论依据不是事实而是信念。
  • 注意力点数：足够的睡眠，适量咖啡因，适时恢复，动用肌肉注意力提升心智注意力 ，有输入才有输出。
  • 时间拆分和番茄工作法。
  • 避免优先级错乱（选择真正紧急的事情排除个人情感因素干扰），避免走进死胡同（坑法则），避免陷入泥潭。

之前试过番茄工作法，在实验室里工作太容易被打断了，可以想办法改进一下。

第十章 预估

  • 专业开发人员只能提供概率预估。
  • 德尔菲法，预估评价的方法，讨论-预估直到意见统一，例如，同时亮手指进行预估。
  • 根据大数定律，可以把大任务分成小人物，分开估计再加总。

第十一章 压力

  • 能回避压力时尽可能回避，慎重承诺，遵循自己的纪律原则。

第十二章 协作

  • 编程就意味着与人协作。

第十三章 团队与项目

  • 组建稳健的团队，让团队在一个个项目中共同工作。

第十四章 辅导、学徒期与技艺

  • 学校传授的是计算机编程理论，但学校无法传授一名编程者所需掌握的原则、实践和技能。
